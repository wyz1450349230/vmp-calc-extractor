# VMP 计算规则提取

## 硬性约束

- 以用户给出的文档、路径、日志、脚本和当前 skill 说明为准。
- 不要发散性搜索无关目录、无关文件、历史工程、网络资料或其它未被用户指定的材料。
- 不要为了“可能有用”去扫大范围文件；只查和当前目标密文、当前日志目录、当前文档明确相关的文件。
- 禁止幻想、禁止猜测、禁止把未验证内容写成结论。
- 找不到来源、区间、计算链或验证结果时，直接标记为“未确认/未找到”，并说明还缺什么证据。
- 所有结论必须能对应到日志位置、trace、handler 地址、计算语句、rw/func/bl 证据或生成脚本验证结果。
- 如果需要扩大搜索范围，必须先说明原因，并限定到具体文件或具体目录；不要默认全盘搜索。
- 平铺优先：按日志执行顺序展开 `v1..vn`，只要加密过程和验证结果正确即可。
- 不要强行找规律、归纳算法、抽象轮函数或重构“漂亮”的通用逻辑；除非用户明确要求，否则不要做归纳。
- 不要把赋值、读取、load/store、slot、内存结构当成主线去追；主线只拿参与目标结果生成的计算指令。

## 启动前先限定材料范围

每次任务开始时，AI 先限定本轮允许使用的材料范围，避免发散：

- 用户明确给出的目标密文、明文、目标值、目标地址或目标 buffer。
- 用户明确给出的日志目录或文件。
- 该目录下和本任务直接相关的 `code` / `func` / `bl` / `rw` 日志。
- 当前 skill 自带的脚本和当前任务中新生成的临时结果。

禁止：

- 不要搜索其它历史工程、旧脚本、网络资料或无关目录。
- 不要默认全盘搜索。
- 不要把历史样例路径写成当前任务依据。
- 如果当前材料不够，直接说明缺少什么，不要靠猜补齐。

## 核心原则

只分析和目标结果有关的 VMP 区间：从已知明文第一次进入 handle 参与计算的位置开始，到目标密文/目标结果被生成、移动、保存或返回的位置结束。

不要分析 VMP 的执行解码、opcode、dispatcher、handler 分发、虚拟 PC 更新、虚拟机框架、无关寄存器状态。区间锁定后，只关注进入 handle 后参与目标结果生成的计算指令链。

## 工作流程

### 1. 锁定有效 VMP 区间

- 起点：明文、key、输入值或其它已知来源第一次进入计算链的位置。
- 终点：目标密文/目标结果被计算出、移动到目标位置、保存或返回的位置。
- 只保留对目标结果有贡献的数据流。
- 丢弃 VM 调度、handler 获取、opcode 解码、栈框架维护、虚拟 PC 变化、防分析噪声等内容。

### 2. 只收集计算指令

主要收集这些指令：

| 指令 | 运算 |
| --- | --- |
| `eor` | `^` |
| `add` | `+` |
| `sub` | `-` |
| `mul` | `*` |
| `and` | `&` |
| `orr` | `|` |
| `lsl` | `<<` |
| `lsr` | `>>` |
| `neg` | `-` |
| `ror` | `ror32` / `ror64` |
| `mov` | 值传递 |

规则：

- `mov` 只有在它传递的值后续参与目标结果生成时才收集。
- 默认不分析条件分支、条件来源、VM 分支行为。
- 默认不分析 `ldr/str`、栈 slot、VM slot、内存临时保存、存储结构。
- `adds/subs/ands` 这类带 flag 的变体，如果出现在计算链里，就按普通 `add/sub/and` 处理，忽略 flag。

### 3. 跟踪符号值，不还原真实寄存器含义

- 维护一个简单的符号表：`计算目的位置 -> vN/表达式`。
- 不需要关心真实寄存器到底是什么值，寄存器只当作表达式临时承载位置。
- 如果操作数已经有符号值，就直接复用已有的 `vN`。
- 如果操作数是外部未知输入，就命名为 `plain`、`key0`、`in_r3` 这类清晰名称。
- 每条被收集的计算指令都生成一个新的临时变量：`v1`、`v2`、`v3`……
- 如果目的位置后面被覆盖，只更新映射关系，不要修改之前已经生成并被引用的 `vN`。

### 4. 先生成粗核对版，不强行找规律

分析时先按收集到的计算指令顺序生成一个平铺的“粗核对版”脚本。目标是还原正确的加密过程，不是归纳漂亮规律；不要一开始就尝试抽象轮函数或寻找通用算法。

处理规则：

- 从上到下只处理计算指令产生的值。
- 能引用已有 `vN` 的地方直接引用。
- 没有匹配到已有 `vN` 的操作数，先保留为日志值、立即数或明确的 `in_xxx`。
- 先让脚本能按当前日志跑通每一步 assert / 片段目标值。
- 再回头看那些“没有变量引用的值”：
  - 是固定常量就标记为常量；
  - 是明文/key/外部输入就改成 seed；
  - 是上一段 VMP 产物就继续分段向上追踪；
  - 找不到来源就标记“未确认”。

粗核对三阶段：

1. **原始日志值版**：先用日志里的 hex 值、立即数、当前可见操作数生成脚本，保证当前日志的 assert 或片段目标值能跑通。
2. **变量替换版**：把能确认来源的 hex 值替换成 `vN`、seed、上一段产物或明确输入。
3. **分段组合版**：当目标密文分段生成时，把各段按实际写入顺序组合成最终结果，再做整体核对。

无变量引用值清单：

粗核对脚本生成后，必须整理没有引用已有 `vN` 的关键值：

- immediate 常量。
- 日志 hex 值。
- `in_xxx` 输入。
- 片段中突然出现但未闭合来源的值。

每个值都要标记来源状态：

- `常量`：日志中稳定出现、无需继续追。
- `seed`：明文、key、外部输入或用户给定值。
- `上一段产物`：来自已经验证的前一段 VMP/独立计算结果。
- `当前段临时值`：当前粗核对中已能由上文计算得到。
- `未确认`：没有证据，不能猜。

表达式保留规则：

- 不要为了可读性主动合并表达式。
- 不要把多行 `vN` 折叠成复杂公式。
- 优先保留和日志顺序一致的 `v1..vn`。
- 只有用户明确要求归纳/简化/提炼通用算法时，才做公式化或轮函数抽象；默认只交付平铺正确的加密过程。

禁止事项：

- 不要为了美化结果提前合并表达式。
- 不要主动追赋值/读取操作当主流程。
- 不要因为某个值像常量或像规律就直接下结论。
- 不要把没有变量引用的值直接幻想成 key、表、状态或上一轮结果；必须结合日志核对。

### 5. 生成计算脚本

按原始执行顺序，每条计算指令生成一行：

```python
v1 = u32(plain ^ key0)      # eor ...
v2 = u32(v1 + 0x39)        # add ...
v3 = ror32(v2, 7)          # ror ...
out = v3
```

规则：

- ARM `w*` 或 32 位日志默认使用 `u32` 包裹。
- 如果日志明确是 64 位，就切换到 `u64` / `ror64`。
- 如果是 byte 级计算，再根据实际情况加 `& 0xff`。
- 输出脚本应包含 `u32/u64` 和 `ror32/ror64` 辅助函数，保证可以直接运行验证。

### 6. 单轮日志确认来源，多轮日志对差异位置矫正

单轮日志处理：

- 先确认目标计算链里每个值的来源基本确定。
- 来源只考虑：
  - 明文 / key / 输入 seed
  - 上一个 `vN`
  - 传递用的 `mov`
  - 立即数常量
  - 其它已收集计算指令的产出
- 生成该轮日志对应的计算脚本。
- 用已知明文/密文对验证。
- 如果不匹配，不要回头分析完整 VMP opcode；只把 mismatch 当作计算链修正信号。

多轮日志处理：

- 对多轮日志重复同样的提取流程。
- 尽量保持一致的 `v1..vn` 编号方式。
- 能按指令地址、handler 位置、日志位置对齐时，优先对齐。
- 找第一处差异位置，重点比较：
  - 期望值和当前生成值
  - 当前源操作数
  - 上一个 `vN`
  - 指令地址 / handler 位置
  - 日志中记录的中间值
  - 位宽 / mask
  - rotate 位数
  - 立即数常量
- 结合日志修正计算链。

常见修正：

- 补上漏掉的值传递 `mov`
- 修正 seed 映射
- 修正 32/64/8 位 mask
- 修正 signed / unsigned 行为
- 修正 `ror/lsl/lsr` 位数
- 修正立即数常量
- 调整起点/终点边界

不要扩展到条件分支、`ldr/str`、VM slot、内存 slot、opcode、执行解码逻辑，除非用户明确要求改变分析范围。

### 7. 目标密文允许分段追踪

目标密文不一定由一个连续 VMP 区间一次性生成。分析时必须允许分段处理：

- 目标密文可能被拆成多段分别生成，例如前 4 字节、后 4 字节、若干 word、若干 byte 分批写入。
- 某一段可能来自当前 VMP 计算链，另一段可能来自独立运算、固定混淆、表运算、已有中间结果或另一段 VMP。
- 一个 VMP 区间结束后，不代表目标密文已经完整结束；如果目标片段来源还没闭合，需要继续向上/向前追踪该片段的来源。
- 如果某个片段的来源又进入了新的 VMP handle 计算链，就继续按本 skill 的规则锁定新的小区间、收集计算指令、生成 `v1..vn`、验证结果。

分段分析流程：

1. 先用 `func/rw` 辅助确认目标密文完整出现的位置、长度和写入顺序。
2. 按实际写入或产出顺序把目标密文拆成片段，例如 `part0/part1/...`。
3. 对每个片段单独锁定“来源参与计算 -> 片段结果产生”的最小 VMP 区间。
4. 每段分别生成计算脚本或公式，并用该片段的目标值验证。
5. 如果片段中间包含独立运算，就把它作为该段的普通计算步骤记录；不要强行合并成一个大 VMP 区间。
6. 如果片段的输入又来自上一层 VMP 产物，则继续向上追踪，直到来源闭合或明确标记“未确认”。
7. 最后把所有片段按目标密文写入顺序组合，并做整体密文验证。

字节序规则：

- 目标密文的字节序必须以日志证据为准。
- 不要默认按大端或小端解释。
- 如果需要按 word 比较，必须同时记录：
  - 原始 byte 序列；
  - 日志中出现的 byte 顺序；
  - 计算产生的 word 值；
  - 最终写入内存的顺序。

分段记录表：

每个片段至少记录这些字段：

| 字段 | 含义 |
| --- | --- |
| `part` | 片段名称，如 `part0` |
| `range` | 目标 byte 范围，如 `bytes[0:4]` |
| `target` | 该片段目标值或 byte 序列 |
| `hit` | 目标值在 func/rw/code 中的证据位置 |
| `vmp_range` | 该片段对应的 VMP trace/handler 区间 |
| `result_var` | 生成该片段的 `vN` 或输出变量 |
| `unknown_values` | 该段没有变量引用且未确认的值 |
| `status` | `已验证` / `部分验证` / `未确认` |

示例格式：

```text
part0: bytes[0:4] = 6e 10 fa d1
来源区间: trace xxx..yyy
结果变量: v1234
验证: 已验证 / 部分验证 / 未确认
未确认值: ...
```

停止条件：

- 当前片段已经能由计算脚本稳定生成，并通过片段目标值验证。
- 或继续向上追踪时没有日志证据。
- 或来源不在用户给定材料范围内。

停止时必须标记状态：

- `已验证`：片段脚本和目标值一致，来源闭合。
- `部分验证`：片段值能对上，但仍有未确认来源。
- `未确认`：片段来源或计算链缺证据。

要求：

- 不要假设“找到一个 VMP 区间 = 整个密文逻辑完成”。
- 不要为了凑完整结果而猜测缺失片段。缺失就标记“未确认”。
- 每个片段都要有对应证据：trace、handler、计算语句、rw/func 命中或脚本验证结果。
- 最终输出既要给整体组合，也要说明每个片段的来源和验证状态。

### 8. 使用 func / bl / rw 日志做辅助核对

func、bl、rw 日志只做辅助核对，不改变主规则：主计算链仍然只从 handle 计算指令里生成。

辅助用途：

- `func` 日志：看 `memmove/memcpy/memset/strlen` 等调用，确认明文输入、目标 buffer、长度、输出位置。
- `bl` 日志：看调用链和关键 call 点，辅助锁定 VMP 前后边界。
- `rw` 日志：看最终内存读写、目标值写入位置、目标 buffer 是否出现预期密文/结果。

使用原则：

- `func/bl` 用来确定“哪一段日志对应这轮输入”和“明文/密文 buffer 在哪”。
- `rw` 用来核对最终结果是否写入预期地址，或者目标值第一次/最后一次出现在哪里。
- 不要因为看到了 `rw` 里的内存读写，就把 `ldr/str`、slot、内存结构加入主计算链。
- 如果生成脚本和目标结果不一致，优先用 `rw` 找第一处目标值缺失/错误的位置，再回到 handle 计算链检查漏掉的 `mov`、常量、位宽、rotate 位数或区间边界。

最终目标：所有选中的日志都能算出对应目标密文/目标结果。最终脚本应该是多轮日志共同矫正后的通用规则，而不是只拟合某一轮日志。

## 抽象参考流程：query + stub 参与加密

下面是从一个 1008 类 a26 生成流程中抽象出来的规范，只保留结构，不绑定任何个人路径或具体样例文件。用于防止分析时发散。

常见参与加密的输入：

- `query`：请求 query 参数字符串。
- `stub`：通常来自 `x-ss-stub`，没有时可能是 16 字节全 0。
- `SM3(query)`：query 字符串 bytes 做 SM3 后的 32 字节结果。
- `SM3(stub)`：stub bytes 做 SM3 后的 32 字节结果。

常见主流程：

```python
query_hash = sm3_hash(query_bytes)
query_part = vmp_handle(query_hash, type=1)

stub_bytes = bytes.fromhex(x_ss_stub) if x_ss_stub else b"\x00" * 16
stub_hash = sm3_hash(stub_bytes)
stub_part = vmp_handle(stub_hash, type=2)

# 后续可能还有一段独立平铺计算：
# - 使用 stub_part 中的若干 vN
# - 使用 query_part 中的若干 vN
# - 做 add/eor/mul/and/or/ror 等普通计算
# - 做少量 byte 级 xor/checksum
# - 最后按顺序拼接 part1/part2/part3/part4
result = part_query_data + part_query_check + part_stub_data + part_stub_check
```

平铺分析时按这个思路拆段：

| 片段 | 常见来源 | 处理方式 |
| --- | --- | --- |
| `part_query_data` | `vmp_handle(SM3(query), type=1)` 返回的 data | 单独锁定 query hash 进入 VMP 到 data 产出的计算链 |
| `part_query_check` | `query_part` 里的若干中间值按 byte 做 xor/checksum | 作为独立平铺计算，不归纳 |
| `part_stub_data` | `vmp_handle(SM3(stub), type=2)` 返回的 data | 单独锁定 stub hash 进入 VMP 到 data 产出的计算链 |
| `part_stub_check` | `stub_part` 里的若干中间值按 byte 做 xor/checksum | 作为独立平铺计算，不归纳 |

注意：

- 这只是结构参考，不代表所有任务都完全一样。
- 不要引用历史文件路径作为依据。
- 当前任务必须以用户给出的日志和目标密文为准。
- 如果日志显示目标密文不是这个结构，就按日志证据处理。
- `query` 和 `stub` 通常都可能参与最终密文，不能只看其中一个。
- 优先参考 `query` 和 `stub/x-ss-stub` 这两个输入的加密路径，但不要排除其它加密参数参与 VMP 计算。
- 如果日志证据显示其它参数、header、body、固定字段、随机数、时间戳或前置结果进入计算链，也要按片段继续平铺分析。
- 其它参数是否参与必须以日志证据为准；不要主动发散搜索，也不要猜测某参数参与。
- `stub` 相关段和 `query` 相关段可能各自有独立 VMP 区间，其它参数也可能有独立区间或参与独立计算段。
- 中间可能穿插普通独立计算，例如若干 `vN` 的 byte xor/checksum；这类也按平铺 `v1..vn` 记录。
- 最终输出要说明每个 part 来自 query、stub、其它参数、独立计算还是未确认。

## 辅助脚本

脚本使用“顶部配置 + 直接运行”的形式，不写命令行参数解析。使用时复制脚本或直接修改脚本顶部配置区；示例不能写死个人电脑路径。

优先使用“两步法”：先收集，再解析。这样可以复用 handler 表，也方便多轮日志重复同一套过滤规则。

### 第一步：收集计算 handle 行

脚本：

```text

```

只改顶部配置区：

```python
INPUT_LOG = Path("/path/to/raw_vmp.log")
```

### 第三步：由 AI 自行结合 func/bl/rw 日志辅助核对

这一步不是让用户运行额外脚本，也不需要产出单独工具。AI 在分析时自行读取用户给定目录中的 `func` / `bl` / `rw` 日志，用来辅助确认：

- 当前目标密文/结果出现在哪些 `func` 拷贝点。
- 明文、密文、输入 buffer、输出 buffer 的地址和长度。
- 目标值在 `rw` 日志里的第一次/最后一次出现位置。
- VMP 计算区间的前后边界是否和 `func/bl/rw` 证据一致。

要求：

- `func/bl/rw` 只作为分析证据，不生成额外辅助脚本。
- 不要让用户自己处理这些日志；由 AI 根据当前任务需要内部查看。
- 不要把 `rw` 内存事件扩展成主计算链。
- 最终回答只汇总必要证据：关键 call、目标 rw 命中位置、用于修正区间或计算链的依据。

主计算链只使用两步法：`vmp_collect.py` 负责收集，`vmp_parse.py` 负责解析。func/bl/rw 辅助核对由 AI 在分析时内部完成，不单独维护辅助脚本。不要再使用命令行参数式的一步脚本。

脚本输出只是第一版结果。后续只围绕计算链、seed 映射、位宽/mask、rotate 位数、立即数常量、区间边界做修正。

## 当前实现的优化点

- 收集和解析拆开：handler 过滤规则可以复用到多轮日志。
- handler 表由当前任务自己收集，直接写入 `HANDLERS = {...}`；不要依赖个人机器上的历史脚本路径。
- 脚本是顶部配置式，不写命令行参数解析。
- 解析阶段读取日志里的 `(r)reg=...` / `(w)reg=...`，可生成每一步实际写回值注释。
- 支持 `ASSERT_TRACE`，运行生成脚本即可定位第一处 mismatch。
- 默认只做“当前寄存器绑定”的数值回溯，避免不同来源碰巧同值时污染计算链；确实需要时再开 `VALUE_BACKTRACK = "global"`。
- `lsl/lsr/ror` 的位数会按 32/64 位自动 mask，避免移位位数异常。
- 未匹配来源可选择输出 hex 常量或 `in_xxx` 输入，方便在多轮日志差异分析时人工矫正。
- AI 分析时可结合 `func/bl/rw` 日志核对区间、buffer、目标值第一次/最后一次出现位置、地址范围最后写入；不要求用户运行额外辅助脚本。

## 输出结果时必须包含

- 选定的起点/终点地址或 marker。
- 收集到的计算指令列表。
- 生成的平铺式 `v1..vn` 粗核对脚本或公式，优先保证加密过程正确。
- 没有变量引用的值列表，以及每个值的来源核对结果：常量 / seed / 上一段产物 / 未确认。
- 需要外部提供的未知输入。
- 单轮日志验证结果。
- 如果目标密文是分段生成，给出每个片段的范围、目标值、出现位置、来源区间、结果变量、未确认值、验证状态和最终组合顺序。
- 如果涉及 word 比较，给出原始 byte 序、日志 byte 序、word 值和最终写入顺序。
- 如果有多轮日志，给出第一处差异位置、差异原因、最终修正内容。
- 如果使用了 `func/bl/rw` 辅助核对，给出关键 call、目标 rw 命中位置、最终用于修正的依据。
- 对无法确认的部分明确写“未确认”，不要补全、猜测或泛化。
